# 项目优化方案

## 一、项目目标

这个项目后续要实现的，不是“把一张照片贴到视频上”，而是做成一条真正像主持人口播的讲解视频。

目标效果是：

- 主持人是画面主视觉
- 背景干净，不抢注意力
- 前期先不强行上字幕
- 先把画面构图做对，再补文字系统

## 二、当前主要问题

目前项目已经可以自动生成视频，但距离“成片感”还有明显差距，核心问题有这些：

- 人物容易像一张被贴上去的矩形照片
- 文字布局过早主导画面
- placeholder 背景仍然影响整体审美
- 动态人物质量还不足以作为默认主模式
- 屏显文案和口播文案耦合太紧

## 三、新方向

后续项目按更高要求来推进，顺序调整为：

1. 先定主视觉模板
2. 先让主持人像主角
3. 先确认静态画面成立
4. 再补动态
5. 最后再补字幕或屏显文案

## 四、主模板规划

### 第一阶段模板

先做一版干净的全屏主持人主视觉：

- 无字幕
- 无底部文案
- 无假信息图
- 无多余边框、线条、辅助标签

要求：

- 主持人画面要成为视觉中心
- 不能再有“贴了一张照片”的感觉
- 背景处理要克制、干净、有成片感
- 这一阶段不强行在画面上塞解释性文字

### 第二阶段模板

等主画面稳定后，再增加文字系统：

- 底部安全区字幕带
- 一次只显示一句短句
- 最多两行
- `narration_script` 和 `screen_script` 完全分离

## 五、主持人策略

### 视觉优先级

主持人是绝对主视觉。

理想素材优先级：

1. 一段干净的正面短视频
2. 一张干净的人像图，再做背景替换或全屏主视觉处理
3. 静态图兜底

### 动态优先级

人物动态建议按下面顺序叠加：

1. 静态高质量主视觉
2. `LivePortrait` 负责轻微头动和面部生命感
3. `MuseTalk` 负责口型同步
4. 再考虑做人脸增强或时序稳定

不要把不稳定的动态输出直接当默认基线。

## 六、文字策略

文字必须拆成两条轨：

- `narration_script`
  给 TTS 用，负责完整口播内容
- `screen_script`
  给屏幕显示用，只保留构图需要的短句

这两者不能完全相同。

在第一阶段里，`screen_script` 可以完全关闭，先不显示。

## 七、实施阶段

### 阶段 A：主视觉基线

交付内容：

- 全屏主持人主模板
- 无字幕
- 无底部文案
- 干净的氛围背景

验收标准：

- 画面不再像贴图
- 没有装饰性噪音
- 主持人像真正的视频主体

### 阶段 B：稳定布局系统

交付内容：

- 可复用的视觉模板参数
- 明确的安全区
- 主持人构图规则
- 背景风格规则

验收标准：

- 各段构图一致
- 主持人和未来文字区不会互相打架

### 阶段 C：屏显文字系统

交付内容：

- 底部短句带
- 句级 `screen_script`
- 跟口播节奏同步的切换逻辑

验收标准：

- 文字可读
- 文字不抢主持人主视觉

### 阶段 D：动态升级

交付内容：

- `LivePortrait` 接入已经确认过的视觉模板
- 条件成熟后接入 `MuseTalk`

验收标准：

- 动态自然
- 没有明显撕裂感
- 没有贴片感

## 十、工作原则

质量提升的顺序应该是：

1. 画面构图
2. 主持人存在感
3. 动态质量
4. 口型同步
5. 字幕系统

如果画面本身还不对，继续堆模型复杂度不会真正解决问题。
